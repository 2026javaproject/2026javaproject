점수 · 스테이지

처치 누적·보스 등장 판정·스테이지 진행

6-1

addKill — 점수·처치수·보스판정 한 번에

적을 잡을 때마다 점수 더하고, 처치수 늘리고, 보스 등장 조건을 확인.

ScoreManager.java
public void addKill(int points) {
    score += points;        // 점수 누적
    killCount++;            // 처치 수 누적
    checkStageThreshold();  // 보스 등장 조건 확인
}

private void checkStageThreshold() {
    int threshold = Math.min(39, 9 + stage);   // 스테이지↑ = 기준↑, 상한 39
    if (!bossSpawnPending && killCount >= threshold) {
        bossSpawnPending = true;   // 신호만 켬(생성은 GamePanel)
    }
}

역할 분리

  • ScoreManager는 보스를 직접 만들지 않고 bossSpawnPending 플래그만 켭니다. 실제 new BossEnemy()는 GamePanel이 담당.
  • 기준치 9 + stage : 스테이지가 오를수록 보스까지 더 많이 잡아야 함. Math.min(39,..)로 상한.
⚠ 기준은 9+stage(=stage1에서 10킬)인데, GameConstants의 배열은 stage1=11로 값이 다릅니다 — 실제 판정은 이 9+stage 기준.
6-2

onBossDefeated — 스테이지 진행

보스를 잡으면 큰 점수를 주고 다음 스테이지로 넘김.

ScoreManager.java
public void onBossDefeated(int bossScore) {
    score += bossScore;          // 보스 보상(1000)
    stage++;                     // 다음 스테이지
    bossSpawnPending = false;    // 보스 대기 해제
}

흐름

  • 일반 적(addKill)과 다른 경로로 보스 보상·스테이지 증가를 처리합니다.
  • stage++ 후 EnemyManager와 checkStageThreshold가 더 어려운 기준으로 동작.
⚠ killCount는 여기서 리셋되지 않습니다. threshold가 누적 killCount와 비교되므로 보스 처치 후 곧 재등장할 수 있어, GamePanel에서 리셋하거나 로직 보완이 필요(발표 시 개선점으로 언급).
6-3

reset / 캡슐화된 상태

모든 진행 상태를 private으로 감추고 getter로만 노출.

ScoreManager.java
private int score, killCount, stage;
private boolean bossSpawnPending;

public ScoreManager() { reset(); }

public void reset() {
    score = 0; killCount = 0; stage = 1; bossSpawnPending = false;
}

public int getScore()  { return score; }
public int getStage()  { return stage; }
public boolean isBossSpawnPending() { return bossSpawnPending; }

캡슐화

  • 상태는 모두 private → 외부가 마음대로 못 바꿈. 읽기는 getter로만.
  • 게임 재시작 시 reset() 한 번으로 모든 값이 초기화돼 일관성을 유지.
  • GamePanel이 isBossSpawnPending()를 매 프레임 확인해 보스 생성 시점을 결정.
